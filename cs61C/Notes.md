## 笔记

# C语言编译过程
写个 simple.c，并且执行gcc simple.c -o simple，：

1. 预处理阶段 (Preprocessing)
最开始这个 .c 文件进入预处理阶段。预处理器进行一些预处理指令的执行(以#为开头的)。它会将头文件展开、纯物理复制并替换过来。
当然，为了避免“无脑复制”导致同一段代码被狂塞好几次（重复定义爆炸），这里必须通过 Header Guards 避免重复定义的情况，例如：

          #ifndef COMMON_H_ (如果没有定义这个宏)
          
          #define COMMON_H_  (那么定义这个宏)
          
          //余下是代码块 (因为cpp不认识这个(因为不是#)所以不是将定义成代码块而是而是声明了一些东西变量，函数等等)
          
          #endif // 结束定义

最后，再进行宏替换，把所有注释擦除。这时候拿到的 .i 文件，依旧只有各种空头支票（声明），没有具体实现逻辑。

2. 编译阶段 (Compilation)
随后呢，便进入编译阶段。编译器把当前预处理过后的纯C代码进行检查。因为头文件只有定义并没有实现，但没关系编译器只看当前文件，看到函数已被定义便觉得完全合法
注意： 此时它会把 C 代码翻译成人类还能看懂的汇编文本指令（.s 文件）。这时候还没有造假地址(这是对于头文件声明但没有具体实现的函数)，留着比如 call add 这样的纯文本符号（名字）。

3. 汇编阶段 (Assembly)
再到汇编阶段，汇编器接手，要把汇编代码翻译成 0101 的机器真正可读的机器码（变成 .o 文件）。
到了这步需要将call add这样的函数调用进行翻译成调用一个地址，如若当前文件有的话将直接计算一个地址偏移量放入没有的话只能在相应的函数调用处留一个全 0 的空白跳转地址(占位符)
不仅如此，它得建个“重定位表”（贴个待办便签），在对应机器码段标识

5. 链接阶段 (Linking)
随后便进入链接阶段。链接器拿着前面留下的便签（重定位表）到处找实现。
无论如何C都会链接libc.so 这种官方库，使用动态链接的方法，链接器在对应缺失实现处，写下一张“名片/欠条”（本质上是 PLT/GOT 表的标识），以此作为以后前往内存 libc.so 库寻找实现的凭证。最终的可执行文件搞定

6. 运行阶段 (Execution)
最后，敲下回车运行。
OS 看到我们程序里留下的那张名片，于是把硬盘上的 libc.so 加载进虚拟内存。而且 OS 极其抠门，因为有“按需调页”机制，用到的才会被装进物理内存，几千个用不到的函数（比如没用到的 malloc）不会加载仍然存于磁盘。
