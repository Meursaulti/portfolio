# C语言编译过程

写一个 `simple.c`，然后执行：

```bash
gcc simple.c -o simple
```

整个过程大致可以拆成下面几步：

---

## 1. 预处理（Preprocessing）

先做一轮“纯文本替换”：

- `#include` → 头文件直接展开
- `#define` → 宏替换
- `#ifdef` → 条件编译
- 注释会被删掉

为了避免头文件被重复 include，需要用 Header Guard：

```c
#ifndef COMMON_H_
#define COMMON_H_

// 声明

#endif
```

这里要注意：

> 这一阶段完全不懂 C 语义，就是在做字符串替换

输出是 `.i` 文件。

---

## 2. 编译（Compilation）

这一阶段才真正“看懂代码”：

- 语法 / 语义检查
- 生成中间表示（IR）
- 做一些优化
- 最后变成汇编（`.s`）

一个关键点：

> 函数可以没有实现，只要有声明就能过编译

比如：

```asm
call add
```

这里还是符号，不是地址。

---

## 3. 汇编（Assembly）

把 `.s` 转成机器码，生成 `.o`

同时做三件事：

- 指令转二进制
- 建 `symbol table`
- 建 `relocation table`

`relocation table`：

> 对于全局变量的读取 / 存储，以及外部函数调用进行标记，留到链接阶段补全。

---

## 4. 链接（Linking）

链接器生成可执行文件：

- 把多个 `.o` 拼起来
- 找函数实现
- 把之前没填的地址补上

分两种：

### 静态链接

- 从库直接拷需要的目标文件一起链接
- 文件变大，但运行独立

### 动态链接（默认）

- 只保留库的标识
- 运行时再加载动态库
- 节省磁盘空间
- 库更新通常不需要重新编译程序

# 第一部分：关键路径（Critical Path）完整流程

同步电路的运行依赖于时钟周期的节奏，数据在寄存器之间传递。

---

## 1. 同步路径概览

```text
        ┌──────────┐
CLK ───▶│ Register A│
        └────┬─────┘
             │
             ▼
      ┌─────────────┐
      │Combinational│
      │   Logic     │
      └────┬────────┘
           │
           ▼
        ┌──────────┐
CLK ───▶│ Register B│
        └──────────┘
```

---

## 2. 时钟上升沿的宏观动作

在每一个时钟上升沿（Rising Edge）触发的瞬间：

- **寄存器 A**：采样并接收当前输入端的新数据 `N`

- **寄存器 B**：采样并接收来自组合逻辑的输出  
  ——这实际上是寄存器 A 在上一个周期接收的数据 `N-1` 经过组合逻辑处理后的结果。

> **核心要求**：寄存器输入必须满足建立时间（Setup Time）与保持时间（Hold Time）约束，否则触发器内部可能进入亚稳态（Metastability），导致输出延迟或逻辑不确定。

---

## 3. 详细流程演进

以下以寄存器 A 的输出为起点进行分析。

### 1. 时钟到 Q 延迟（t_pcq）

时钟上升沿到来后：
- 主锁存器关闭，内部捕获的数据已经稳定（前提是满足 t_setup / t_hold）
- 从锁存器随即打开，主锁存器的状态开始驱动从锁存器内部节点
- 若数据值与从锁存器原先保持的值不同，从锁存器发生翻转
- 翻转后的状态经过一段路径最终稳定的呈现在 Q 端

这一系列内部传播过程所需要的时间称为 **t_pcq（Clock-to-Q delay）**

---

### 2. 组合逻辑处理（t_comb）

信号离开寄存器 A 后，进入组合逻辑网络。

组合逻辑的延迟由：

- 逻辑门延迟
- 导线传播延迟
- 扇出负载
- 晶体管驱动能力

等共同决定。

信号经过逻辑运算后，到达寄存器 B 输入端所需的时间称为：

```text
t_comb
```

这通常是整条同步路径中最大的延迟来源。

---

### 3. 门前数据建立（t_setup）

处理后的信号到达寄存器 B 输入端后，必须在下一个采样边沿到来之前提前稳定。

这一段必须提前稳定的时间称为：

```text
t_setup（建立时间）
```

建立时间存在的根本原因是：

```text
触发器内部需要足够时间，
使内部状态已经明显偏向某个稳定逻辑态
```

这样当采样边沿到来、内部正反馈结构开始主导状态时，系统才能可靠地收敛到确定逻辑值。

如果输入变化过晚，则在采样瞬间，内部节点可能仍接近中间平衡状态，从而导致亚稳态（Metastability）。

---

## 4. 关键路径延迟公式

为了保证系统正常工作，一个时钟周期 `T` 必须能够覆盖整条数据路径的传播时间：

```text
T ≥ t_pcq + t_comb + t_setup
```

如果时钟周期小于该延迟，则在下一个时钟边沿到来时：

- 新数据可能尚未完成组合逻辑传播
- 或寄存器内部尚未建立稳定状态
- 导致建立时间违例（Setup Violation）

此时触发器可能输出错误结果，甚至进入亚稳态。

---

# 主锁存器与从锁存器

为什么寄存器需要建立时间（Setup Time）与保持时间（Hold Time）？

这需要深入到主从 D 触发器（Master-Slave D Flip-Flop）的内部。

---

## 1. 为什么需要建立期（t_setup）？

寄存器的核心是由主锁存器（Master Latch）和从锁存器（Slave Latch）串联组成的。

### 主锁存器透明阶段

当主锁存器处于透明状态时（具体对应哪个时钟电平取决于实现方式），外部输入可以传播到主锁存器内部节点。

### 建立倾向（t_setup 窗口）

在采样边沿到来之前的一小段关键时间内，外部输入必须保持稳定。

这并不只是“给电容充电”，更重要的是：

```text
让锁存器内部节点已经明显偏向某个稳定状态
```

例如内部节点已经形成：

```text
A≈0V
B≈1V
```

或：

```text
A≈1V
B≈0V
```

这样当内部正反馈闭合时，系统才能快速、可靠地收敛到确定逻辑状态。

如果输入在采样边沿前变化过晚，内部节点可能仍接近中间状态（如两侧都接近 `0.5V`），从而导致亚稳态（Metastability）。

---

## 2. 为什么需要保持期（t_hold）？

### 锁存切换瞬间（采样边沿来到）

当采样边沿来到时，主锁存器逐渐关闭，同时内部反馈电路（由交叉耦合反相器组成）开始主导内部状态。

### 再生放大

如果内部节点已经具有明确倾向，正反馈会迅速完成再生过程，将状态收敛到稳定逻辑电平。

例如：

```text
0.1V → 0V
0.9V → VDD
```

### 防止污染（t_hold）

由于真实电路中的传输门与时钟信号都存在有限延迟，锁存器不会在采样边沿瞬间完全隔离输入。

因此，在采样边沿之后的一小段时间内：

```text
输入仍必须保持稳定
```

否则新的输入变化可能继续传播进主锁存器，扰动尚未完全稳定的内部状态，从而导致保持时间违例（Hold Violation）。

---

## 3. 数据交接与输出

与此同时，主锁存器与从锁存器的透明状态发生切换：

1. **从锁存器动作**：从锁存器进入透明状态，允许主锁存器中的状态向输出方向传播。

2. **物理传递**：如果主锁存器已经稳定到确定逻辑状态，其输出会强力驱动从锁存器内部节点，使从锁存器迅速收敛到相同逻辑值。

3. **Q 端稳定**：经过有限传播延迟后，输出端 `Q` 呈现出合法、稳定的新逻辑电平。

---

## 4. 周期循环

当时钟进入下一相位后：

- 主锁存器重新进入透明状态
- 从锁存器关闭并保持当前数据
- 电路准备接收下一轮输入数据


---

# 五级流水线数据通路

单周期数据通路的缺点在于一个周期当中的不同组件处于空转状态，并没有充分使用硬件，导致 CPU 速度并不快，吞吐量并不高。由此衍生出了流水线（Pipeline）的概念，通过流水线寄存器将单周期划分为几个阶段，使得同一个周期所有组件都能被不同的指令充分利用。

## 五级流水线架构

流水线分为五个 Stage（阶段），每个阶段由流水线寄存器划分：

- **IF (Instruction Fetch)**：取指并更新程序计数器 (PC)。

- **ID (Instruction Decode)**：译指、读取寄存器文件以及生成立即数。

- **EX (Execute)**：执行阶段，处理算术逻辑运算和分支比较。

- **MEM (Memory)**：读取内存或者写入内存。

- **WB (Write Back)**：写回寄存器文件。

### 流水线寄存器

- **IF/ID**：保存指令以及 PC。

- **ID/EX**：保存读取的寄存器数据、立即数、PC、指令。

- **EX/MEM**：保存 ALU 的输出、rs2 数据、PC、指令。

- **MEM/WB**：保存 ALU 的输出、MEM 的输出、PC+4、指令。

---

# 指令执行流程示例

假设有五条指令 ${I_1, I_2, I_3, I_4, I_5}$（以下都满足建立时间）：

## 1. 第一个时钟上升沿：I₁ 进入流水线

- 来到 IF 阶段，等待时钟到 Q 的时间，PC 输出指令地址进入 IMEM。等待读取延迟后，指令来到流水线寄存器；同时 PC 也会输出到相应的流水线 PC 寄存器，并且输出到加法器加四，经过 MUX 延迟来到下一个 PC 寄存器。

---

## 2. 第二个时钟上升沿：I₂ 进入流水线

- I₂ 来到 IF 阶段并重复 I₁ 在 IF 的流程。

- I₁ 来到 ID 阶段，等待时钟到 Q 的时间，指令被送入寄存器文件开始读取。同时 PC 继续传播，指令传入控制单元、立即数生成器以及对应的指令寄存器。经过一定的延迟生成控制信号（如 ImmSel），随后立即数生成器输出至相应的流水线寄存器（通常寄存器文件的输出会稍晚一点到达流水线寄存器）。

---

## 3. 第三个时钟上升沿：I₃ 进入流水线

- I₃ 来到 IF 阶段，I₂ 来到 ID 阶段。

- I₁ 来到 EX 阶段，同样经过时钟到 Q 的时间，PC、rs2、指令继续传播。指令进入控制单元生成对应的控制信号；rs1 和 rs2 的比较结果传入控制单元，经过延迟生成 PCSel 传入 IF 段的 MUX（这也是后续控制冒险的来源）。同时根据控制信号决定 PC / rs1 / rs2 / imm 谁通过 MUX 进入 ALU 进行运算，经过一定延迟后来到 ALU 流水线寄存器。

---

## 4. 第四个时钟上升沿：I₄ 进入流水线

- I₄ 来到 IF，I₃ 来到 ID，I₂ 来到 EX。

- I₁ 来到 MEM 阶段，经过时钟到 Q 的时间，PC 继续传播，指令继续传播。控制单元决定是读取还是存储内存，经过一定的读取延迟后来到对应的 MEM 流水线寄存器。

---

## 5. 第五个时钟上升沿：I₅ 进入流水线

- I₅ 来到 IF，I₄ 来到 ID，I₃ 来到 EX，I₂ 来到 MEM。

- I₁ 来到 WB 阶段，经过时钟到 Q 的延迟后，将指令送入控制单元以决定 PC+4、ALU 输出、MEM 输出谁通过 MUX，以及是否写入寄存器文件。
在一个周期中，不同的指令依靠这些寄存器同步执行，指令在每个阶段被独立译码生成对应的控制信号。

## 总结

可见一条指令的执行时间变得更长了（需要五个周期，且每个周期都有流水线寄存器的时钟到 Q 延迟等开销），但是**吞吐量变高了**。

在第五个周期开始（即四个周期后），此后的每一个周期结束都能输出一条指令。

由于我们将任务切分，时钟周期大幅缩短，周期的长度取决于耗时最长阶段的**关键路径时间**（通常是 ALU 或 MEM），否则会造成最小周期违例。

---

# 流水线冒险 (Hazards)

流水线在提高频率和吞吐量的同时，也引入了冒险。冒险分为三类：结构冒险、数据冒险、控制冒险。

---

## 1. 结构冒险 (Structural Hazard)

硬件资源抢夺。例如在 IF 阶段和 MEM 阶段，不同的指令需要同时访问内存。

### 解决方式

增加硬件资源。例如采用哈佛架构（分离的指令存储器和数据存储器）来避免内存访问冲突。

---

## 2. 数据冒险 (Data Hazard)

前后指令存在数据依赖关系。例如：

```assembly
add s0, t0, t1
add s0, s0, s0
```

前文流程可知，`s0` 需要在第一条指令的 WB 阶段后才会被写回；而第二条紧随其后的指令，在 ID 阶段读取寄存器文件时，第一条指令才堪堪到达 EX 阶段。此时第二条指令会拿到错误的旧值进行计算。

### 数据冒险的解决思路

#### 1. 停顿 (Stall / Bubble)

引入气泡，使得第二条指令在第一条写回后再执行。

加入气泡指的是：检测到冲突后，将 PC 寄存器的写使能改为 0，IF/ID 流水线寄存器的写使能设为 0（保持当前状态），同时将 ID/EX 寄存器的控制信号清零（相当于插入一条 `addi x0, x0, 0` 即 NOP 指令）。

但这会使得 CPI (Cycles Per Instruction) 变高，性能下降。

---

#### 2. 寄存器半周期读写 (Double Pumping)

支持在 WB 阶段的前半周期写入，ID 阶段的后半周期读取。这样可以少停顿一个周期。

只要：

```text
控制信号延迟 + MUX延迟 + 建立时间 + 写入时间 + 读取时间
```

在一个时钟周期内即可支持。

---

#### 3. 数据转发 (Data Forwarding / Bypassing)

不用老老实实等待写回完毕。

假设第一条指令在 MEM 阶段（第四周期），第二条指令在 EX 阶段，此时只需从 EX/MEM 的 ALU 结果寄存器拉出一条导线，配合检测逻辑和 MUX，直接将结果作为当前 ALU 的输入即可。

如果是第三条指令和第一条指令冲突，则从 WB 阶段将MUX的输出数据拉回 EX 阶段。

### 转发检测逻辑（EX 级与 MEM 级转发）

```c
// EX 级冒险转发 (Forwarding from EX/MEM to EX)
if (EX_MEM.RegWrite && EX_MEM.rd != 0 && EX_MEM.rd == ID_EX.rs1)
    ForwardA = 10;

if (EX_MEM.RegWrite && EX_MEM.rd != 0 && EX_MEM.rd == ID_EX.rs2)
    ForwardB = 10;


// MEM 级冒险转发 (Forwarding from MEM/WB to EX)
// 注意：需要确保不会覆盖更近的 EX 级转发

if (MEM_WB.RegWrite && MEM_WB.rd != 0 &&
    !(EX_MEM.RegWrite && EX_MEM.rd != 0 &&
      EX_MEM.rd == ID_EX.rs1) &&
    MEM_WB.rd == ID_EX.rs1)
{
    ForwardA = 01;
}

if (MEM_WB.RegWrite && MEM_WB.rd != 0 &&
    !(EX_MEM.RegWrite && EX_MEM.rd != 0 &&
      EX_MEM.rd == ID_EX.rs2) &&
    MEM_WB.rd == ID_EX.rs2)
{
    ForwardB = 01;
}
```

---

### Load-Use 冒险（无法通过纯转发解决）

```assembly
lw s0, 0(a1)
add s0, s0, s0
```

对于 `lw` 指令，数据要在 MEM 阶段结束才能从内存读出，因此无法在第一条指令的 EX 阶段结束时把数据转发给第二条指令的 EX。

必须**停顿一周期**，然后再使用转发。

### Load-Use 停顿检测逻辑（在 ID 阶段检测）

```c
if (ID_EX.MemRead &&
   (ID_EX.rd == IF_ID.rs1 ||
    ID_EX.rd == IF_ID.rs2)) {

    // 触发 Stall：
    // 保持 PC 和 IF/ID，
    // 将 ID/EX 清空为气泡
}
```

---

## 3. 控制冒险 (Control Hazard)

分支指令导致的执行流不确定。

```assembly
beq t1, t2, lab
add s0, t0, t1
add s0, s0, s0

lab:
sub s1, s0, s3
```

决定是否跳转是在 EX 阶段（或优化后的 ID 阶段），此时流水线已经取入了后续指令。

如果分支发生跳转，流水线里已经进入的指令就是错误的。

### 控制冒险的解决思路

#### 1. 停顿 (Stall)

遇到分支就停顿，直到算出结果，但太慢。

---

#### 2. 静态分支预测 (Static Branch Prediction)

固定预测分支“不跳转”（总是执行下一条）。

如果不跳转，毫无停顿；如果发生了跳转，则需要**冲刷 (Flush)** 掉流水线上已经取入的错误指令。

具体做法和气泡类似，使用 MUX 将 IF/ID 和 ID/EX 的指令替换为 `nop` (`addi x0, x0, 0`)，但**不需要**锁死 PC 的写使能，因为要跳到新的目标地址。

---

#### 3. 动态分支预测 (Dynamic Branch Prediction)

处理器在运行时根据分支指令的历史执行行为来预测其跳转方向。

通过维护分支历史表等硬件结构记录过往结果，提前取指。这种方法能更准确地适应程序的实际运行模式，显著提升处理器性能。

---

## 性能铁律 (Iron Law of Performance)

衡量一段程序执行时间的公式：

$$
\text{CPU Time} =
\frac{\text{Instructions}}{\text{Program}}
\times
\frac{\text{Clock Cycles}}{\text{Instruction}}
\times
\frac{\text{Seconds}}{\text{Clock Cycle}}
$$

也可以简写为：

$$
\text{Time} =
\text{Instruction Count}
\times
\text{CPI}
\times
\text{Clock Cycle Time}
$$

- **Instruction Count (IC)**：取决于算法、编程语言、编译器和指令集架构 (ISA)。

- **CPI (Cycles Per Instruction)**：取决于计算机的微架构设计（如流水线结构、缓存设计）。

- **Clock Cycle Time (时钟周期时长)**：取决于硬件实现和半导体工艺制程。

---

## 效能铁律 (Iron Law of Power / Energy)

衡量处理器能耗的核心公式分为能量和功率两个维度。

### 1. 单程序能耗 (Energy per Program)

$$
\text{Energy} =
\frac{\text{Instructions}}{\text{Program}}
\times
\frac{\text{Energy}}{\text{Instruction}}
$$

通过优化指令数或采用低功耗硬件设计来减少每次运算消耗的焦耳。

---

### 2. 动态功耗公式 (Dynamic Power)

$$
\text{Power} =
\text{Capacitance}
\times
\text{Voltage}^2
\times
\text{Frequency}
$$

即：

```text
P = C * V² * f
```

降低电压 ($V$) 是最有效的降功耗手段，同时降低频率 ($f$) 也能减小瞬时发热，但会影响上述性能铁律中的时钟周期。

# 缓存 (Cache) 机制与底层运行逻辑

随着计算机的发展，CPU 的主频越来越高，内存的读取速度也在提升，但两者的增长速度截然不同：前者增长卓越，后者则相对滞后。这就导致了一个核心问题：**CPU 在执行读取或写入操作时，会造成极大的时钟周期停顿。**

看两个直观的数字：CPU 的时钟周期约为 1ns，而访问一次内存需要 100ns，这意味着单次访存就需要停顿上百个周期。如果使用的是超标量处理器(现代处理器大多都是)，一个周期能同时发射多条指令，那么这将会造成几百上千条指令的停滞，现代处理器面临的停滞将更加夸张。

为了解决这个问题，缓存机制被引入了。

由于物理结构和庞大数据量的限制，DRAM（内存）的速度很难做得很快，也无法直接集成到 CPU 内部。因此，业界设计了一种距离 CPU 更近、容量很小，但数据读取速度极快的存储结构，那就是缓存。

## 缓存是怎么作用的？

缓存将通过复制主内存的一部分数据到其自身，可以将其理解成一个中间人。它提前将你想要的数据拿到了，这样你就可以不用跑个老远去主内存访问。

这很显然，缓存需要尽可能准确地预测你可能需要的数据——毕竟它那么小，主内存那么大，如何预测？这就依托于**局部性原理**了。

显然上面有一个问题，**需要怎么样知道缓存里有你需要的数据呢？**

先看看**缓存行 (Cache Line)** 最基本的组成部分：**数据块 (Data Block)**，**tag**，**offset**。

- **数据块**由字节组成。
    
- **tag** 就是该数据块在主内存的高位地址部分。
    
- **offset** 自然就是低位。
    

**为什么不将 offset 放到 tag 当中呢？**
<img width="458" height="281" alt="屏幕截图 2026-06-01 201047" src="https://github.com/user-attachments/assets/2bf2fa12-e66f-44ab-ba67-fa1b3758b205" />
来看看这张图，如果 offset 和 tag 合并了，一个**缓存行**是不是就变成了只能容纳一个字节的**数据块**(若是如此空间局部性就消失了见下文)呢？一个字节组成的数据块太零散了，这可能会造成多次缓存的访问。而且 offset 和 tag 分离不仅可以在一个**数据块**当中以一个字节的精度选择，还可以节约比特位。

看看现在我们有了 tag，offset，数据块。

如果说现在有一条读取指令：

`lb a0, 0(0x43F)`

首先我们将 `0x43F` 分成：

- **tag :** `0b0100 0011 11`
    
- **offset:** `0b11`
    

我们是不是就可以利用传来缓存的地址，并将其拆分成 tag 和 offset？

将其 tag 去和缓存中所有 tag 相比较，如果比较成功了，那么这就称为**缓存命中 (Hit)**？不还是不是见下文。

在去比较的对应的**数据块**同时被送入 MUX（多路选择器），并且以 offset 为 Sel（选择信号），然后在确认命中后将其送出。

如果没有标签匹配，那么就称为**缓存不命中 (Miss)**。此时缓存就要去往主内存，前往 `0x43C`（访问主内存以 4 的倍数去访问，所以这里进行往最近的 4 倍数地址取）读取整个**主存块**大小（这里是四字节），并写入到**缓存行**（随便去哪一行都行吗还是有什么策略呢？）中的数据块部分，并且写入相对应的 tag，然后再进行 offset 读取返回给 CPU。

以上就是一个通过缓存来访问内存的一个大致流程，但并不标准，还得完善完善。

### 硬件层面：比较器与有效位 (Valid Bit)

**想想我们真的需要一个一个 tag 的去比较吗？有 tag 真的足够了吗？**

显然在硬件当中我们是不需要线性的去比较的，而且**有 tag 也并不足够**。

先讲讲为什么有 tag 并不足够？

在计算机当中删除一个文件，其实并不是说真正的去将那个内存地址的数据抹除为 0，而是再也没有一个东西指向他了，那个内存也不再被视为已被占用了，我们的缓存自然也是如此。

假设你电脑重启之后运行你自己写的一个小脚本，在其中有一些访问内存的操作，正巧你需要访问的内存地址在缓存中存在。可这真的是你想要的数据吗？可能真的会是你想要的，但也很可能是垃圾数据。为了计算机运行的稳定，是不可能放任这种不确定的，所以在缓存行的标签段又多了一个**有效位 (Valid Bit)**，以 0 为无效，以 1 为有效。

**线性的比较？**

No, No。这可是在硬件层面上，我们完全可以为输入的标签和缓存中的所有标签同时比较。只需要为每一个缓存行加一个比较器。输入 tag 与缓存行现有的 tag 共同输入比较器得出结果。

但在连接到或门之前，必须先结合有效位做判断。因为有效位是每个缓存行私有的，所以比较器的输出结果必须先与该缓存行自身的有效位（Valid Bit）进行一次与运算（AND）。只有当 `Tag 匹配成功` 且 `Valid 为 1` 时，该行的与门才输出 1。最后，将所有缓存行的与运算结果共同连接到一个或门（OR）。如果任意一行满足条件，或门输出 1，这才是真正的 Hit。

好了，如果完善以上的这两个要点就已经得到了一个**全相联缓存**。后续还有直接映射缓存，组相联缓存，全相联不是够好了吗还要那两个做啥？硬件成本太高了，几百上千个比较器，CPU 融化了……但还是有应用，只不过场景不多。

## 缓存的核心——局部性原理

在继续之前先了解一下缓存的核心——局部性原理：

- **时间局部性：** 如果某个数据项被使用了，那么它很有可能在未来的一段时间再次被使用。
    
    - _一个栗子：_ 名为 `sum` 的变量在求和的仪式当中会被多次访问——是读取也可能是写入。
        
- **空间局部性：** 如果某个数据项被使用了，那么它的邻居也很有可能被访问。
    
    - _一个栗子：_ 名为 `array` 的数组正在被程序员使用遍历求和。
        

局部性原理是缓存的核心。如果这两个原理不成立的话，那么缓存存储什么就变得不再明确，其效能就没有那么出色了。后面的设计也是围绕着这个原理。

## 缓存设计策略与映射方式

同时再看看缓存设计策略，这可以引申出有意思的问题：

- **放置策略：** 一个**主存块**可以放在缓存的什么位置？这个**主存块**可以关联到哪个**缓存行**？
    
- **标识：** 如果一个**主存块**在缓存中，如何找到它？标识与放置策略密切相关。
    
- **替换策略：** 发生未命中时，应替换哪个**缓存行**？
    
- **写入策略：** 写入时会发生什么？
    

在以上我们接触到的一种缓存叫全相联缓存，而其既然是一个缓存方式，必然有对应缓存设计策略的问题回答。

### 1. 全相联缓存 (Fully Associative Cache)

- **放置策略：** 一个**主存块**可以放置在缓存中的任意**缓存行**中。记得吗，全相联通过给每一个**缓存行**的 tag 都配备一个比较器，所以不需要考虑一个**主存块**需要放到哪里，因为放到哪里都不影响。
    
- **标识:** 是的没错，就是那个 tag。
    
- **替换策略:** 有三种可以选择，根据具体的应用场景做适配各有优劣：
    
    1. **LRU (最近最少使用):** 替换最近最少使用的**缓存行**。这很好的践行了时间局部性，也是使用最多的一个替换策略。缺点就是要实现排行哪些**缓存行**是最少使用，在硬件上实现比较复杂。
        
    2. **FIFO (先进先出):** 先进先出，这可能不会遵循时间局部性。例如只有三个**缓存行**容量的缓存，访问序列 `0x00`, `0x01`, `0x00`, `0x02`, `0x00`, `0x03`。这样 `0x00` 明明是被访问最多的却被优先剔除了，但它也能做到 LRU 的合理近似。
        
    3. **Random (随机):** 随机地替换一个**缓存行**。优点是易于实现，缺点是不稳定，并且高度依赖场景。
        
- **写入策略:** 如果要写入呢？应该怎么办？这里分直写和写回两种方法：
    
    - **直写 (Write-through - 和主内存时刻保持同步):** 写入的时候如果命中就直接写入缓存行并写到主内存。没命中呢？通常是直接写入主内存，不调入缓存（这叫做不按写分配 No-write allocate）。缺点是每一次写入都要花费较多的时间，会导致性能下降；优点是未命中惩罚相比写回会轻得多。
        
    - **写回 (Write-back - 缓存的数据较为崭新):** 这是最常用的写入策略。在写入的时候，如果命中了，只写入缓存行，直到被替换的时候再写入主内存。很明显对吧，这个策略可以将频繁写入、频繁修改的成本大大降低，可能几百次写入就只需要一次的写回主内存。如果没命中，就先将对应主内存的数据写入缓存中再执行上述操作，这也叫写分配 (Write allocate)。为了区分哪些是在被替换的时候需要执行写回操作的**缓存行**，这里给 tag 部分加上了脏位 (Dirty bit)。如果为 1 需要写入，如果为 0 无须写入。缺点是未命中惩罚较重，缓解这个缺点的一个方法是设计一个写缓冲区，具体我不赘述。
        

### 2. 直接映射缓存 (Direct-Mapped Cache)

直接映射缓存的缓存方式比起全相联的直接更多一些内敛，但也僵硬。

我们的内存地址现在不止能够解读成 tag，offset 了，而是能够解读成 **tag, index, offset**。让我们更详细的来看看吧：

- **tag** 依然不变。
    
- **offset** 是块内偏移，但也能够看作一个**主存块**有多少字节。
    
- 而我们的 **index** 则是给**主存块**分位置的同时，也是找位置的。
    
- **放置策略:** 全相联缓存能够将一个**主存块**放入任意**缓存行**，需要大量比较器，硬件成本较高。直接映射缓存则选择了一种更加简单的方案：**每个主存块只能映射到唯一的缓存行。**
    
    那么问题来了：如果给定一个**主存块**，如何快速确定它应该放到哪一行呢？最直接的做法是：`缓存行号 = 主存块地址 % 缓存行数`。
    
    例如缓存有 4 行：**主存块** 0 去第 0 行，**主存块** 1 去第 1 行……**主存块** 4 又去第 0 行。不同的**主存块**地址会周期性地映射到同一行。在硬件中实现取模电路成本高，因此通常将行数设计成 $2^n$。对于二进制数，$x \mod{2^n}$ 等价于取 x 的最低 n 位。所以硬件根本不需要实现复杂的取模运算，只需要从地址中直接取出对应的几位作为索引（Index）即可。
    
    **为什么不能直接对完整地址取模？**
    
    原因在于，一个**主存块**通常包含多个字节，地址的最低几位实际上用于表示块内偏移量（Offset）。如果把 Offset 也参与索引计算，同一个**主存块**中的不同字节可能会被映射到不同**缓存行**，这显然是不正确的。所以地址最终被拆分为：`| Tag | Index | Offset |`。
    
- **标识:** 同样是 tag，但是这次找的方式需要利用 index。由上得来我们知道每个**主存块**对应的 index 是唯一的，所以我们在读取的时候可以直接利用分线器得到 index，随后再去对应的 index 处所对应的 tag 进行比较……如果 tag 相同那么命中，不相同那么 miss。
    
- **替换策略:** 先前说其僵硬的原因正是在此，因为其只能取模长然后去到相应的位置。这意味着缓存空间还有，却因为规则只能去特定的一个座位，就算那里已经有人了。所以这很容易造成时间局部性的削减，甚至有时候没有时间局部性可言。
    
    想象我要写一个复制逻辑，而刚好原数组和目标数组的内存地址恰好取模后是同一个地址，数组的长度也相同。来推演一下：首先访问原数组的第一个元素发生强制未命中，写入目标数组的第一个位置发生未命中写分配，此时因为替换逻辑原数组对应的**缓存行**被替换又没了，原数组第二个元素的访问 miss，目标数组的写入又 miss，复制完了也没有 Hit 过。这就是一个典型的例子，叫做**乒乓效应 (Ping-Pong Effect)**。
    
- **写入策略:** 通用，不再赘述。
    

### 3. 组相联缓存 (Set-Associative Cache)

这是最常用的缓存方式，同时他也能描述全相联和直接映射。组相联是一种权衡：直接映射冲突 miss 发生的概率很大，全相联的硬件成本太高。组相联平衡了这两点。

- **放置策略:** 其 tag 沿用直接映射的布局，但是 index 索引到的不再是一个单一的**缓存行**，而是一个包含 n 个**缓存行**的组。可以叫做 n 路组相联，依然是通过取模计算索引以此放置。因为不再是单一的**缓存行**而是多行汇聚成的一个组，在组中可以像全相联一般放置。由此，1 路组相联其实就是直接映射缓存；当 组数 = 1 时，M 路组相联也可以看作是一个全相联。
    
- **标识:** 依旧 tag，通过输入的地址得到组索引。如果是 n 路组相联将会有 n 个比较器，以此支持对应组的**缓存行**能够并行的进行比较。电路逻辑和全相联类似。
    
- **替换策略:** 先是通过分线得到索引去到相应的组，然后在组内可以使用上面所述的所有替换策略（不包括直接映射那种强制占座）。
    
- **写入策略:** 通用，不再赘述。
    

## 未命中分类 (Miss Classification)

- **强制未命中 (Compulsory Miss):** 缓存初始化，也叫冷启动。在这个时候呢，所有缓存行的有效位都是 0，都是不可用的。所以每一个被访问的**主存块**都无法避免的要经历一次。当然还是有办法降低强制未命中的数量和未命中占比，但是无法改变每一个**主存块**都需要经历一次的事实。
    
- **非强制未命中:**
    
    - **冲突未命中 (Conflict Miss):** 在直接映射中最为直观，也就是上面的乒乓效应。明明还有位置，但是非要去抢占导致冲突，这就是冲突未命中。
        
    - **容量未命中 (Capacity Miss):** 在全相联来看最为直观。所有**缓存行**都被占用了不得不替换，被替换掉的那个可能在接下来一段时间又被访问了，这就是容量未命中，由容量所造成。
        

有时候冲突未命中和容量未命中会不太好分辨。一个好的方式是：将其看成一个等价容量的全相联缓存，会不会导致未命中呢？如果不会的话，那么就是冲突，否则就是容量。

## 缓存优化与 AMAT 公式

以上的缓存优化，大多主要围绕这个计算机体系结构中的核心评估式子：

$$AMAT = \text{Hit Time} + \text{Miss Rate} \times \text{Miss Penalty}$$

- **AMAT (平均访存时间):** 衡量存储系统性能的终极指标，越小越好。
    
- **Hit Time (命中时间):** 也就是去缓存里找数据，并成功把数据拿出来的时间。
    
- **Miss Rate (未命中率):** 缓存没命中次数占总访存次数的比例。
    
- **Miss Penalty (未命中惩罚):** 没命中时，去下一级缓存/主存把数据搬回来的漫长等待时间。
    

对其中任何一个项进行改善，大多会损害到其他项，这就是硬件设计里的 Trade-off（权衡）：

1. **提高缓存行大小 (Line Size / Block Size):** 可以提高空间局部性（降低 Miss Rate），也可以减少上述的强制未命中占用的未命中比例以及数量，但同时可能会损害时间局部性。因为提高了**缓存行**大小，缓存的**行数**同时就减少了，**缓存行**当中的数据多样性也就变小了。一个极端的例子，一个**缓存行**直接占用了所有缓存的容量，这样空间局部性极高但是时间局部性极低，乒乓效应又来了。同时，提高**缓存行**大小也会因为一次需要传输更多的数据，增加了传输的时间（提高了 Miss Penalty）。
    
2. **提高缓存容量 (Cache Size):** 简单粗暴，能装更多东西，极大地降低了容量未命中（降低 Miss Rate）。但缺点是，容量越大的 SRAM，其内部寻址、信号路由经过的门电路越多，会导致命中时间变长（提高 Hit Time），也相应地增加了制造成本。
    
3. **更高的相联度 (Associativity):** 从直接映射变成 4 路、8 路组相联，可以有效减少乒乓效应，降低冲突未命中（降低 Miss Rate）。但代价是，组内的多路并行比较需要更多的比较器和更复杂的 MUX，这会拉长电路延迟（轻微提高 Hit Time）。
    
4. **多级缓存 (Multilevel Caches):** 这是现代 CPU 的标配。在 CPU 和主存之间塞入 L1、L2 ，L3 缓存。L1 做得极小极快，专门为了压低 Hit Time；而 L2/L3 做得容量比较大，用来做主存和 L1 之间的缓冲，专门用来拦截未命中请求，从而大大降低去主存取数据的 Miss Penalty。
    
5. 写入缓冲区 (Write Buffer): 在缓存和主存之间放置的一个小容量队列，用于掩盖向主存写入数据造成的停顿（降低 Miss Penalty）。其具体时序取决于写入策略：

	**直写 (Write-through) 场景：** CPU 会将数据同时写入缓存和 Write Buffer，随后立刻去执行下一条指令，由 Buffer 在后台负责将数据缓慢写入主存。
    
	**写回 (Write-back) 场景：** CPU 在正常运行时仅将数据写入缓存（并标记为脏位）。直到缓存发生未命中，该脏块面临被替换（Eviction）时，硬件的缓存控制器才会将这块脏数据丢进 Write Buffer，并立刻去主存拉取新数据。CPU 自身几乎感受不到脏块淘汰写回带来的停顿。

# 性能编程与并行化
   
# Flynn's Taxonomy

先介绍一个很重要的分类。

对串行硬件和并行硬件进行分类。

**Flynn's Taxonomy**：

- **SISD**：单条指令处理单个数据，一种串行计算机，在指令流或数据流中都不利用并行性。
    
- **SIMD**：单条指令对多个数据进行操作。其能够利用同一条指令同时处理多个数据元素，从而显著减少指令数量，具有数据级并行性（Data-Level Parallelism）。
    
- **MISD**：多条指令对单个数据进行操作，应用场景十分少见，有着并行性。
    
- **MIMD**：多个自主的核心同时在不同的数据上执行不同的指令，有着并行性。
    

这里我们会详细讲讲 SIMD 和 MIMD。

## SIMD (单指令多数据流)

严格来说，SIMD 是一种计算模型，而在 x86 架构中，它体现为一系列指令集扩展（如 SSE、AVX 等）。

其核心能力是：

**通过单条指令对多个不同的数据进行并行操作。**

这依托于 CPU 内部独特的宽寄存器硬件构造。

### 寄存器位宽演进

在架构演进过程中，SIMD 寄存器位宽曾出现过 128 位、256 位以及 512 位（早期尚有 64 位等更小位宽，但不作为重点）。

现代处理器已经有了 512 位宽寄存器（AVX-512），同时为了保持向后兼容，依然保留了对较小位宽（如 128 位 SSE、256 位 AVX2）操作的支持。

根据具体的指令，我们可以将这些宽寄存器拆分成若干个较小的数据元素，然后对这些元素同时进行相同的操作。

### 举个栗子：数组对应元素相加

假设我们有以下数组：


```c
int nums[8] = {0, 1, 2, 3, 4, 5, 6, 7};
```

**目标**：

将前 4 个元素 (0,1,2,3) 与后 4 个元素 (4,5,6,7) 对应相加，结果存入新数组。

**预期结果**：


```c
result[4] = {4, 6, 8, 10};
```

#### 方案 A：常规 C 语言标量写法


```c
int result[4];

for (int i = 0; i < 4; i++) {
    result[i] = nums[i] + nums[i + 4];
}
```

在传统标量代码中，CPU 需要重复执行加载、计算、地址计算、循环控制以及分支跳转等操作。

当数据规模不断增大时，这些额外开销会逐渐累积，从而限制程序性能。

#### 方案 B：使用 SIMD 内置函数（Intrinsics）

我们可以使用编译器提供的内置函数（可以看作是用 C 语言语法去直接映射底层的 SIMD 汇编指令）：


```c
#include <immintrin.h> // 引入 SIMD 头文件

// 1. 加载前 4 个 int (0,1,2,3) 到 128 位寄存器 temp1
__m128i temp1 = _mm_loadu_si128((__m128i*) &nums[0]);

// 2. 加载后 4 个 int (4,5,6,7) 到 128 位寄存器 temp2
__m128i temp2 = _mm_loadu_si128((__m128i*) &nums[4]);

// 3. 向量化加法：4 对 32 位整数同时相加
__m128i sum = _mm_add_epi32(temp1, temp2);

// 4. 将 128 位结果一次性存回内存
int result[4];
_mm_storeu_si128((__m128i*) result, sum);
```

执行时可以简单理解为：

`temp1 = [0, 1, 2, 3]`

`temp2 = [4, 5, 6, 7]`

执行加法后：

`sum = [4, 6, 8, 10]`

SIMD 的优势主要来自于利用宽寄存器中的多个计算通道（Lane），在一次指令发射中同时完成多个数据元素的运算
还有一个附带的优势减少了指令数量

在传统标量代码中，CPU 需要重复执行加载、计算、循环控制以及分支跳转等操作；而 SIMD 能够将多个数据元素的计算合并到少量向量指令中完成。

此外，宽寄存器也能够一次加载和存储更多数据，进一步减少数据搬运所需的指令数量。

在本例中，原本需要多次独立执行的加法操作，被压缩为了少量向量指令完成，因此整体执行效率得到了提升。

## MIMD (多指令多数据流)

SIMD 利用的是单个核心内部的数据级并行。

然而随着处理器频率提升逐渐遭遇功耗墙，同时摩尔定律也不再像过去那样持续带来性能红利，仅依靠单个核心已经难以继续获得显著性能提升。

于是处理器厂商开始增加核心数量，让多个核心同时参与计算。

现代 CPU 往往包含多个能够独立执行程序的核心。

这些核心可以同时处理不同的数据，并执行不同的指令。

这种通过多个核心同时工作的方式，被称为**线程级并行（Thread-Level Parallelism，TLP）**。

而在 Flynn 分类法中，其对应的并行模型便是：**MIMD（Multiple Instruction Multiple Data）**。

让我们直观地看看其为什么有效。

假设我们有以下数组：

C

```
int nums[8000];
```

**目标**：将数组中的所有元素累加到变量 sum 当中。

#### 方案A：常规 C 语言标量写法


```c
int sum = 0;
for (int i = 0; i < numsSize; i++) {   
    sum += nums[i];
}
return sum;
```

这是一个完全串行的计算过程，每次循环只能处理一个数据。

#### 方案B：SIMD 写法


```c
// 将128位寄存器设置为全0
int temp_arr[4] = {0};
__m128i temp_vec = _mm_setzero_si128();

// 在现实中无法保证 numsSize 是4的倍数
// 因此需要进行尾部处理
for (int i = 0; i < numsSize / 4 * 4; i += 4) {
    temp_vec = _mm_add_epi32(
        temp_vec,
        _mm_loadu_si128((__m128i *)(nums + i))
    );
}

_mm_storeu_si128((__m128i *)temp_arr, temp_vec);

// 尾部标量处理
int tail_sum = 0;
for (int i = numsSize / 4 * 4; i < numsSize; i++) {
    tail_sum += nums[i];
}

return temp_arr[0] + temp_arr[1] + temp_arr[2] + temp_arr[3] + tail_sum;
```

相较于标量代码，每次循环能够同时处理 4 个 32 位整数，并且需要执行的指令数量显著减少。

#### 方案C：MIMD 写法

首先介绍一个可移植、标准化的并行编程 API —— OpenMP。

在 C 语言中，OpenMP 使用称为 `pragma` 的编译器指令。

OpenMP 之所以采用这种机制，是因为不支持 OpenMP 的编译器会直接忽略这些 pragma 指令，因此同一份代码既可以在支持 OpenMP 的环境下并行运行，也能够退化为串行程序正常编译执行。

### Fork-Join 并行模型

OpenMP 采用的是 Fork-Join 并行模型。

当程序开始运行时，操作系统会创建一个进程，并在其中创建主线程。

当执行到并行区域时，主线程会创建（或唤醒）多个工作线程共同执行任务-fork。

当并行区域结束后，所有线程重新同步，程序继续以单线程方式执行-join。

通常情况下，在并行区域外定义的变量会被多个线程共享，而在并行区域内部定义的局部变量则会成为线程私有变量。

来看一个 4 核机器上的例子：


```c
#include <stdio.h>
#include <omp.h>

int main() {
    int nums[8000] = {...};
    int sum = 0;
    int numsSize = 8000;

    // OpenMP 将循环迭代划分给多个线程执行
    #pragma omp parallel for reduction(+:sum)
    for (int i = 0; i < numsSize; i++) {
        sum += nums[i];
    }

    printf("%d\n", sum);
    return 0;
}
```

这看起来可能会快得多，让我们看看发生了什么：

- Thread0 -> `nums[0~1999]`
    
- Thread1 -> `nums[2000~3999]`
    
- Thread2 -> `nums[4000~5999]`
    
- Thread3 -> `nums[6000~7999]`
    

每个核心只需要处理原来四分之一的数据。看起来：

`1 Core -> T ; 4 Cores -> T / 4`

那速度能够提升四倍吗？

## 阿姆达尔定律 (Amdahl's Law)

一个很常见但是陷阱的期望：**期望计算机某一方面的改进能使得整体性能按改进幅度成比例提升。**

但阿姆达尔定律告诉了我们这一心碎的事实，并不能，并且极其受到串行的代码部分影响。

先看看这个改写的公式：

$$S = \frac{1}{(1 - p) + \frac{p}{s}}$$

- $S$：整体加速比（Speedup）
    
- $p$：可并行化的代码比例（frac_optimized）
    
- $s$：并行部分的加速因子（factor_improvement）
    

假设我们上面的式子串行部分 10%，并行 90%，并且加速因子为 4，代入公式可得加速比约为 3.07 倍。

假设我们有**无限个核心**将并行部分消除，加速比极限即为 10 倍。无限的核心在此也仅能达到 10 倍的速度。

假设另一个程序串行部分 50%，并行 50%。即便拥有无限个核心，加速比极限也仅为 2 倍。

以上的例子告诉我们，硬件做努力还不够，编写软件的程序员也同时要尽可能地编写串行占比更少的程序。

## 数据竞争与原子锁

其实在多线程当中会有数据竞争的问题，而在上面我们通过 OpenMP 提供的预处理指令解决了：

`#pragma omp parallel for reduction(+:sum)`

来看下面这个将会导致数据竞争的版本：


```c
#include <stdio.h>
#include <omp.h>

int main() {
    int nums[8000] = {...};
    int numsSize = 8000;
    // 共享资源
    int sum = 0;

    // OpenMP 将循环迭代划分给多个线程执行，但未处理共享变量的归约
    #pragma omp parallel for
    for (int i = 0; i < numsSize; i++) {
        // 数据竞争
        sum += nums[i];
    }

    printf("%d\n", sum);
    return 0;
}
```

什么是数据竞争呢？

**不同的线程访问同一个共享资源，并且至少有一个是写操作，它们接连发生。**

举个栗子，一个线程在一次循环中大致会执行如下汇编操作：

代码段

```c
// 从内存读取 sum
lw t0, 0(a0)

// 从数组读取 nums[i]
lw t1, 0(s0)

// 执行加法
add t0, t0, t1

// 写回 sum
sw t0, 0(a0)
```

设两个线程交错执行：

代码段

```c
Thread0:
lw t0, 0(a0)    // 读取 sum=0

Thread1:
lw t0, 0(a0)    // 读取 sum=0

Thread0:
lw t1, 0(s0)    // nums[0]=5
add t0, t0, t1
sw t0, 0(a0)    // 写入 sum=5

Thread1:
lw t1, 0(s1)    // nums[1]=7
add t0, t0, t1
sw t0, 0(a0)    // 写入 sum=7
```

问题出现了：我们的预期是 `sum = 12`，而实际 `sum = 7`。

两个线程都认为自己拿到的是最新的数据，最终导致其中一个更新被覆盖，这也叫**丢失更新**。

需要如何解决这个问题呢？加锁？来看看：

我们设置一个变量 `lock`，`lock = 1` 即为锁上，`lock = 0` 即为未锁。


```c
// 锁区
while(lock == 1) {
    // wait
}
lock = 1;

// 代码区
sum += nums[i];

// 解锁
lock = 0;
```

这看似可行，但**抱歉还不行**。至少现在没有在高级语言层面能实现真正的锁。为什么不行？

两个线程可以在锁状态改变之前，同时读取到锁为 0。两个线程都认为没锁，于是数据竞争又发生了。

要解决这个问题，我们要回到汇编的层面。

本质上，这个锁会发生问题，在于“读取”和“修改”是分布执行的两条指令。如果我们能让核心使用**单条指令**完成读取和修改呢？

问题解决了，利用单指令的**原子性**。

以 RISC-V 为例：

代码段

```c
# acquire
        li             t0, 1
# 获取s0处的旧数据，将其赋值给t0,同时将t0的数据写入s0处
try:    amoswap.w.aq   t0, t0, (s0) 
# 比较t0是否不为0
        bnez           t0, try
locked: # already locked now

# release
unlock: amoswap.w.rl   x0, x0, (s0)
```

无论两个线程的指令怎么交错，原子指令都能保证在“锁到解锁”的区间内，只准许一个线程访问。

在底层硬件中，原子指令的写入操作会触发总线仲裁或硬件保护（例如排他性访问）。同一时刻只能有一个线程成功夺取写入权，被裁决失败的线程会发现自己未成功获取锁，只能waiting...这便是**原子锁**。

## 缓存一致性 (Cache Coherence)

这样我们便能够通过锁的方式来解决数据竞争。而这一切其实隐藏着一个前提：**缓存一致性（Cache Coherence）**。

我们在上面的例子当中是在一个多核多线程的语境当中。还记得吗？每一个核心是有着独立的缓存的。

如果两个核心缓存了同一个内存地址的副本，如若 CPU1 写入了该地址，CPU2 缓存对应的地址内容没有改变，这就是缓存不一致。

而我们的系统正常运转需要缓存一致性。

**定义**：

缓存一致性是指在多核或多处理器系统中，确保所有处理器对共享数据的读写操作遵循统一的顺序和可见性规则，使得任何一个核心对共享内存的修改都能被其他核心及时且正确地观察到。

在 CS61C 当中对于缓存一致性实现没有做过多的讨论，在这里我也仅讲讲抽象层面上的一种方式。

维护缓存一致性我们采用的方法是使用缓存一致性协议，这里讲讲经典的 **MOESI 协议**。

MOESI 是一种完整的缓存一致性协议，它描述了缓存块在核心间的生命周期流转。摒弃生硬的概念，让我们以一场实际的交互来看看它是如何运作的：

- **独占 (Exclusive)**：CPU1 访问一个地址 `x`，发生 Cache Miss。它通过内存控制器访问 DRAM 拿到该地址的数据放入缓存。此时其他核心尚不拥有该地址的有效副本，CPU1 独自占有它，且与内存保持一致（干净的最新数据），这就是独占。
    
- **共享 (Shared)**：接着，CPU2 访问同一个地址 `x`，同样 Miss。此时数据可以由内存提供，也可以由当前的独占者 CPU1 提供。此时副本不再由 CPU1 一人独占，CPU1 从独享状态变成共享者（S）状态，拿到数据的 CPU2 自然也是（S）。
    
- **修改 (Modified)**：随后，CPU2 打算向地址 `x` 写入一个新数据。在此之前，它必须先发送失效信息（Invalidate），使得包括 CPU1 在内的其他核心中该地址的缓存行被设为无效（I）。随后 CPU2 的状态变成修改（M），此时它才具备写入权限并正式写入。此时 CPU2 处于修改（M），CPU1 变为无效（I）。
    
- **拥有 (Owned)**：CPU1 又需要读取这个地址，此时发生了一致性缺失（Coherence Miss）CPU1 准备向内存请求该地址，但这个请求会被处于 M 状态的 CPU2 **直接拦截**。CPU2 会把手里的最新脏数据转发给 CPU1。转发完成后，CPU2 降级为拥有（O）状态（它保留对该脏数据的主导权，无需立即写回主存），而拿到数据的 CPU1 变为共享（S）状态。这就实现了高效的“脏共享”。
    
- **无效 (Invalid)**：在刚才 CPU2 准备写入数据时，CPU1 的缓存行被强行踢出局的状态，就是无效（I）。
    

> 引用 CS61C 课程笔记的一个段落：
> 
> 所有者（Owner）：拥有最新数据，其他缓存可能存有副本（它们必须处于共享状态）。该缓存是多个持有缓存行有效副本的缓存之一，但拥有对其进行修改的独占权。它必须将更改广播给共享该行的所有其他缓存。引入所有者状态允许数据的脏共享，即修改后的缓存块可以在不同缓存间移动而无需更新主内存。

### 伪共享 (False Sharing)

在结合了 MIMD 和缓存一致性协议（如 MOESI）后，会产生一个经典的性能隐患——**伪共享**。

假设两个独立的线程分别频繁修改变量 A 和变量 B。逻辑上它们没有数据竞争。但如果 A 和 B 在内存中紧挨着，被加载到了**同一个 64 Byte 的 Cache Line** 中。

由于缓存一致性协议的粒度是 Cache Line，当线程 0 修改 A 时，会导致线程 1 所在核心的整个 Cache Line 变为 Invalid；线程 1 修改 B 时，又会导致线程 0 的失效。

这个 Cache Line 会在两个核心之间疯狂传递失效信息（Ping-Pong 效应），导致极大的性能损耗。

## 性能编程 (Performance Programming)

性能编程就是将以上的一系列融合，多线程配合上 SIMD，再加上循环展开以及编译器的优化。

还有尽可能地在软件层面**提高缓存命中率**，这相当重要。性能瓶颈有时候就来自于缓存命中率太低。

### 缓存分块 (Cache Blocking)

举个栗子：矩阵乘法。

假设我们有两个矩阵：

C

```c
double A[N][N];
double B[N][N];
double C[N][N];
```

计算 $C = A \times B$ 最直观的写法如下：


```c
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        for (int k = 0; k < N; k++) {
            C[i][j] += A[i][k] * B[k][j];
        }
    }
}
```

时间复杂度: $O(N^3)$。似乎没什么问题？

不，问题在于缓存。

CPU 并不是每次只加载一个数据。例如一个 64 Byte 的 Cache Line，一次可以加载 8 个 `double`。

而 C 语言的二维数组采用 **Row-major（行优先）** 存储：

Plaintext

```
B[0][0] B[0][1] B[0][2] ...
B[1][0] B[1][1] B[1][2] ...
B[2][0] B[2][1] B[2][2] ...
```

连续的数据位于同一行。然而观察内层循环这句：`B[k][j]`

对于固定的 `j`，我们实际上是在按列访问：`B[0][j], B[1][j], B[2][j]...`

这意味着：**CPU 刚刚加载进来的 Cache Line 很可能只用了其中一个元素，随后马上访问一个新的 Cache Line，导致大量缓存失效。**

换句话说：

`Memory -> Cache -> 使用一次 -> 丢弃`

CPU 大量时间都花在了等待内存上。

于是前人们想到：既然缓存很宝贵，能不能让已经加载进缓存的数据被反复利用？这便是**缓存分块（Cache Blocking）**。

我们不再一次处理整个矩阵，而是把矩阵切成很多小块（Block）：

```c
┌────┬────┐
│ A0 │ A1 │
├────┼────┤
│ A2 │ A3 │
└────┴────┘
```

每次只计算其中一个小区域：

```c
for (int ii = 0; ii < N; ii += BLOCK)
    for (int jj = 0; jj < N; jj += BLOCK)
        for (int kk = 0; kk < N; kk += BLOCK)
            
            // 针对 Block 内部进行矩阵乘法
            for (int i = ii; i < ii + BLOCK; i++)
                for (int j = jj; j < jj + BLOCK; j++)
                    for (int k = kk; k < kk + BLOCK; k++)
                        C[i][j] += A[i][k] * B[k][j];
```

为什么要切成特定大小的 Block 呢？这里的 BLOCK 大小并不是随意的，它需要被精心设计。为了彻底避免数据在反复计算的过程中互相将对方驱逐出去（避免容量缺失，Capacity Miss），我们需要确保在内层的计算过程中，**A、C 小分块的一行数据，以及 B 这个小分块能够被完整地装进 Cache 中**。

这样做的好处在于：某一小块 A、B、C 数据被加载到缓存后，在被驱逐出去之前会被反复使用很多次。
数据复用率显著提高。算法复杂度并没有变化，仍然是 $O(N^3)$，但运行速度可能提升数倍甚至十几倍。

让缓存得到更好的利用，在提高性能上是一个绝佳的方向。

### 循环展开与函数内联

还可以指定我们会反复访问的变量（比如说一个求和 sum 变量），将其设置成寄存器，这样就可以再次减少内存的访问。但一般来说编译器比我们手动优化的要好。

以及**循环展开**，提高每一次循环的步骤数而减少分支跳转判断的数量：

C

```c
int i = 0;
for (; i < max/4*4; i+=4) {
    arr[i] = f(i);
    arr[i+1] = f(i+1);
    arr[i+2] = f(i+2);
    arr[i+3] = f(i+3);
}

for (; i < max; i++) {
    arr[i] = i * i;
}
```

但这会使汇编指令更多，生成的执行文件也要更大，而且还会使得代码难以阅读。通常不需要我们动手，交给编译器！

以及**函数内联（Inlining）**，在函数很短且大范围调用的话，我们可以将函数直接展开，以此减少函数调用时序言（Prologue）和尾声（Epilogue）中压栈出栈的额外开销。这请交给编译器！

以上交给编译器的部分需要在我们编译时输入参数优化等级。

> 引用图灵奖得主 Don Knuth 的一段话：
> 
> “许多当今的软件工程师所共有的传统观点主张在小处忽略效率；但我认为，这不过是对他们所看到的那些精打细算却因小失大的程序员所进行的滥用行为的一种过度反应，这些程序员既无法调试也无法维护他们“优化”过的程序。...
> 
> 毫无疑问，追求效率的圣杯会导致滥用。程序员会把大量时间浪费在思考或担心程序中非关键部分的速度上，而当考虑到调试和维护时，这些对效率的追求实际上会产生强烈的负面影响。我们应该忘掉那些微小的效率问题，大约 97% 的时间都应该如此：过早优化是万恶之源。
> 
> 然而，我们也不应错过那关键的 3% 中的机会。一个优秀的程序员不会被这样的推理所麻痹而自满，他会明智地审视关键代码；但只有在这些代码被识别出来之后才会这样做。”

以上就是性能编程与并行化的所有内容了。
