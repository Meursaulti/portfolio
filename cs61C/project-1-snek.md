# Snek

**类别：** C 语言基础 / 内存管理  

---

## 1. 项目概述

本项目是一个基于 C 语言实现的命令行贪吃蛇游戏。项目基于一门课程给定的初始骨架代码展开，要求实现游戏中核心的底层逻辑。

该项目整体并不算复杂，具备 C 语言基础即可着手。就是你不再有高级语言的轻松了!

## 2. 具体实现细节

在既定框架的接口约束下，我主要通过组合多个辅助函数，完成了以下三大核心模块：

* **棋盘解析与初始化 (`load_board` / `build_board`)**：
  处理从外部文件或标准输入（stdin）加载不同规格的文本棋盘。为了应对未知长度的数据流，实现了一套基于 `realloc` 的动态扩容缓冲区，将读取到的文本流映射为 C 语言的二维字符数组。
* **蛇的位移与交互逻辑 (`update_game` 及相关辅助函数)**：
  将蛇的运动拆解为对二维数组中具体字符的读取与覆写。
  * **坐标预测**：利用 `next_square` 结合当前蛇头的方向（如 `W/A/S/D`），计算下一步的坐标内容。
  * **状态改变**：通过 `head_to_body` 和 `body_to_tail` 等函数，在蛇移动时动态转换字符（例如将表示身体的 `^` 转换为表示尾巴的 `w`），从而在终端上模拟出蛇的滑行效果。
  * **碰撞判定**：判断下一格是墙壁（`#`）、食物（`*`）还是自身的身体，进而触发增长或死亡（`death`）逻辑。
* **内存回收闭环 (`free_game`)**：
  编写了针对动态二维数组的精确清理逻辑。通过遍历行释放列内存，最后释放二维指针与结构体本身，确保程序在退出时能干净地交还所有堆内存。

## 3. 代码优化与 Bug 修复

在实现基础逻辑后，我针对代码的健壮性与执行效率进行了几项关键排查与修复：

* **剥离隐式性能开销**：原代码曾将求取字符串长度的 `strlen()` 调用直接写在 `for` 循环的判断条件中，导致每次循环迭代都会产生 O(N) 的遍历开销。改进至将长度计算缓存至循环外，消除了多余的性能损耗。
* **异常分支的资源清理**：增添了对堆内存分配失败和重分配失败的处理逻辑，对于堆内存分配失败后释放先前分配的所有内存避免内存泄漏，对于重分配用临时指针接收避免永久丢失原内存块，以及对于文件指针的漏关进行了修复，秉持“谁打开谁关闭”的原则
* **指针与数组越界防御**：修复了在初始化多条蛇时，由于漏写结构体数组索引（直接向基地址写入数据）导致的内存覆盖 Bug。规范了指针算术与数组偏移量的使用，确保所有访问都在合法的内存边界内。

## 4. 学习与反思

* **单一职责与模块化思维**：项目中包含了十几个如 `is_tail`、`get_next_row` 这样极小粒度的辅助函数。这种设计让我再次体会到，将复杂的判定逻辑下放给底层小函数，不仅能让主更新逻辑 (`update_game`) 变得像读文章一样清晰，也极大降低了 Debug 的难度。
* **内存管理**：每一次内存请求（`malloc`）和资源获取（`fopen`）都有失败的可能，要处理这些失败的情况不能相信堆中一定有内存，输入一定合法，在这次实现中，失败路径处理比主逻辑更容易出问题，尤其是在 realloc 和文件读取阶段。
