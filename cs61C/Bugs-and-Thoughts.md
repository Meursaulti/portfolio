# 🐛 Bug 复盘记录

**日期：** 2026-04-26
**项目：** CS61C Project 1 (Snake)
---

### 1. 现象观察 
* **错误类型：** OOB Write (越界写入) 导致的 Memory Corruption (内存踩踏)。
* **预期行为：** `create_default_game` 返回的棋盘，在坐标 `(0, 0)` 处应为字符 `#`。
* **实际表现：** 测试框架抛出 `Assertion error: at (row 0, col 0), expected # but got P`。控制台视觉输出正常，但底层断言失败。

### 2. 最小复现代码
以下代码剥离了业务逻辑，单纯复现内存的践踏过程：
```c
#include <stdlib.h>
#include <stdio.h>
#include <string.h>

int main() {
    int row_num = 18;
    
    // 💥案发现场：错误使用了 sizeof(char)
    char **board = malloc(sizeof(char) * row_num); 

    for (int r = 0; r < 5; r++) { 
        // 申请堆内存
        board[r] = malloc(22); 
        // 写入正常业务数据
        strcpy(board[r], "####################\n");
        
        // 观察底层状态：每次循环后，检查 board[r]是否有被正确写入
        printf("分配 board[%d] 后, board[0][0] 的字符是: %c\n", r, board[0][0]);
    }
    return 0;
}
```
*(注：在某些系统的 `ptmalloc` 连续分配策略下，当跑到 `r=4` 时，由于指针的写入越界，`board[0][0]` 的输出会瞬间从 `#` 变成乱码或特定的内存地址字节(实际显示:"P\367UUUU"))*

### 3. 分析
* **内存状态推演：**
  在 64 位系统中，指针 `char *` 占用 8 字节。需要分配 18 个指针，物理需求应为 $18 \times 8 = 144$ 字节。但代码写 `sizeof(char)`（1 字节），实际仅分配 18 字节。`ptmalloc` 考虑到 16 字节对齐，分配了一个 32 字节的 Chunk。
  
  堆分配器默认在相邻地址连续分配内存，例如先前给予二维数组所分配是32字节，设其范围是0x1000-0x101F,如若0x1020往后恰好满足所需字符串大小那么将在这里分配`board[0]`便指向这里并且写入`####################\n0`。后续将r = 1, r = 2, r = 3 写入后 这 32 字节内存便耗尽，后续写入 `board[4]`, 这个指针是将存储在0x1020这不是合法的访问还记得之前`board[0]`是指向0x1020的吗原先写入的数据被踩踏了!。
  
  这 8 字节的堆地址，以小端序直接砸进了紧挨在后面的 `board[0]` 所指向的字符串所在地址。ASCII 码的 `0x50`（字符 `P`）应当是某个堆内存地址的组成部分，从而无声无息地覆盖了原本合法的 `#`。系统未抛出 Segfault，因为越界仍发生在该进程分配的合法虚拟内存页内(真难找啊我记得我写了加了star的呜呜)。

### 4. 调试路径
1.  **在代码内加入打印语句** 打印写入后的每一行确认是否是写入了错误的字符，得到正确的输出排除写入逻辑问题，既然不是写入逻辑能是什么呢？
想到当前打印是依托于刚写入完状态，并不排除后续被别的数据覆盖，使用GDB进行调试对指定数据加入探针watch
3.  **GDB 内存追踪：**
    * 获取地址：`print &board[0][0]`
    * 植入探针：`watch *<实际地址>`
    * 触发点：程序在 `board[4] = malloc(...)` 暂停，指出该行代码写入了 8 字节指针，破坏了目标字符。

### 5. 解决方案
1. 给予二维字符数组的每一行分配正确的空间，`char **board = malloc(sizeof(*board))`

### 6. 总结优化

使用结构性的debug方法而非如同无头苍蝇，寻找更多信息以缩小问题范围
使用内存检查工具AddressSanitizer或者Valgrind能快速找到内存方面的问题，本质上这个问题并不复杂，
